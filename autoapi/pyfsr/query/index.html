pyfsr.query¶

Fluent builder for the FortiSOAR query DSL.

FortiSOAR’s ad-hoc query endpoint (POST /api/query/{module}) takes a body of the shape:

{
  "logic": "AND",
  "filters": [
    {"field": "severity.itemValue", "operator": "eq", "value": "Critical"},
    {"logic": "OR", "filters": [ ... ]}        # nested group
  ],
  "sort": [{"field": "createDate", "direction": "DESC"}],
  "__selectFields": ["uuid", "name", "severity"],
  "limit": 30
}

Query builds that dict so callers don’t hand-assemble it. Every mutator returns self for chaining:

Query().eq("status.itemValue", "Open").gt("createDate", ts).sort("createDate").limit(50)

The builder validates operators and value-arity against the operator knowledge base (pyfsr.query_models). Pass module= to additionally validate field paths against the shipped field/relationship KB (pyfsr.fields):

Query(module="alerts").eq("severity.itemValue", "Critical")  # field path checked

The validated, typed form is available via Query.model() (a QueryBody); Query.to_body() renders the dict.

Fluent builder for a FortiSOAR query body.

Parameters:

  • logic – Top-level logical join for this group’s filters ("AND" or "OR").

  • module – Optional module name; when set, field paths are validated against the shipped field/relationship knowledge base.

>>> Query().eq("severity", "High").to_body()
{'logic': 'AND', 'filters': [{'field': 'severity', 'operator': 'eq', 'value': 'High'}]}
where(field: str, operator: str, value: Any = None) → Query[source]¶

Add a raw leaf filter. Prefer the named helpers (eq, in_ …).

eq(field: str, value: Any) → Query[source]¶

Match records where field equals value.

For picklist fields use the .itemValue sub-path to compare by display name, or the bare field name to compare by IRI:

Query().eq("severity.itemValue", "High")   # friendly name
Query().eq("status.itemValue", "Open")
Query().eq("uuid", "3f2a...")               # exact UUID
neq(field: str, value: Any) → Query[source]¶

Match records where field does not equal value.

Query().neq("status.itemValue", "Closed")  # exclude closed alerts
lt(field: str, value: Any) → Query[source]¶

Match records where field is less than value.

Typically used with epoch timestamps:

import time
Query().lt("createDate", time.time() - 86400)  # older than 24 h
lte(field: str, value: Any) → Query[source]¶

Match records where field is less than or equal to value.

gt(field: str, value: Any) → Query[source]¶

Match records where field is greater than value.

Typically used with epoch timestamps:

import time
Query().gt("createDate", time.time() - 3600)  # last hour
gte(field: str, value: Any) → Query[source]¶

Match records where field is greater than or equal to value.

in_(field: str, values: list[Any]) → Query[source]¶

Match records where field equals any value in values.

Query().in_("severity.itemValue", ["Critical", "High"])
Query().in_("uuid", ["3f2a...", "7b1c..."])
nin(field: str, values: list[Any]) → Query[source]¶

Match records where field equals none of values.

Query().nin("status.itemValue", ["Closed", "Resolved"])
like(field: str, value: str) → Query[source]¶

Case-insensitive substring match on field.

The server interprets value as a SQL LIKE pattern (% is wildcard), but a plain string also works as a contains check:

Query().like("name", "phishing")        # name contains "phishing"
Query().like("sourceId", "SIEM-%")      # starts with "SIEM-"
notlike(field: str, value: str) → Query[source]¶

Case-insensitive substring non-match on field.

Inverse of like():

Query().notlike("name", "test")  # exclude records named "test*"
contains(field: str, value: Any) → Query[source]¶

Match records where a collection field contains value.

Used for to-many relationship fields (tags, indicators, owners) — checks that at least one related item matches:

Query().contains("recordTags", "malware")
exists(field: str, value: bool = True) → Query[source]¶

Match records where field is present (True) or absent (False).

Query().exists("assignedTo")           # record has an assignee
Query().exists("assignedTo", False)    # record is unassigned
isnull(field: str, value: bool = True) → Query[source]¶

Match records where field is null (True) or non-null (False).

Query().isnull("resolvedDate")          # not yet resolved
Query().isnull("resolvedDate", False)   # has a resolved date
changed(field: str) → Query[source]¶

Match records whose field changed (trigger-condition operator).

Value-less — only meaningful inside a playbook start/update trigger filter.

in_all(field: str, values: list[Any]) → Query[source]¶

Match records whose field contains all of values.

Trigger-condition operator (distinct from in, which is any-of).

between(field: str, low: Any, high: Any) → Query[source]¶

Match records where field is between low and high (inclusive).

Convenience operator that compiles to a gte + lte pair under the hood. For example, between("createDate", start_ts, end_ts) is equivalent to composing gte and lte leaves in an AND group:

Query().between("createDate", 1700000000, 1700086400)
Parameters:

  • field – The field to filter on.

  • low – The lower bound (inclusive).

  • high – The upper bound (inclusive).

related(path: str, operator: str, value: Any = None) → Query[source]¶

Filter by a field in a related module using dot-walk syntax.

This is a convenience helper for relationship-traversal filtering. Under the hood, it’s equivalent to using where() with a dot-separated or double-underscore-separated path, but it makes intent explicit:

# These are equivalent:
Query().related("alert.status.itemValue", "eq", "Open")
Query().where("alert.status.itemValue", "eq", "Open")
Query().where("alert__status__itemValue", "eq", "Open")

The FortiSOAR platform treats . and __ identically in field paths, and adding a relationship hop (e.g. alert) narrows scope via an inner join on the related module.

Parameters:

  • path – A dot-separated or double-underscore-separated path traversing related modules. The final segment is the field to match.

  • operator – The comparison operator ("eq", "gt", etc.).

  • value – The value to compare (arity determined by operator).

Returns:

self for chaining.

Examples:

Query(module="incidents").related("alerts.severity.itemValue", "eq", "Critical")
Query().related("assignedTo.name", "like", "alice")
group(query: Query) → Query[source]¶

Nest another Query as a sub-group (its own logic + filters).

or_(query: Query | None = None) → Query[source]¶

Create or nest an OR-logic group for ergonomic composition.

This is a convenience wrapper around Query(logic="OR").group(...) that lets you build OR groups inline without explicit Query("OR") construction:

Query().eq("status", "Open").or_(Query().eq("type", "A").eq("severity", "High"))

Equivalently, nest an empty OR group to be populated inline:

(Query()
 .eq("status", "Open")
 .or_()
 .eq("type", "A")
 .eq("severity", "High"))
Parameters:

query – Optional pre-built Query to nest as an OR group. If None, creates a fresh OR context for inline building.

Returns:

If query is provided, returns self (for chaining). Otherwise, returns a fresh OR-context proxy that appends to this Query when a terminal method (to_body/model) is called.

and_(query: Query | None = None) → Query[source]¶

Create or nest an AND-logic group for ergonomic composition.

The AND counterpart to or_(). Use it to build an explicit AND sub-group — most often as one branch inside an outer OR, where the sub-group’s “all of these” semantics must be isolated from the parent:

# (status == Open) OR (type == A AND severity == High)
(Query("OR")
 .eq("status", "Open")
 .and_()
 .eq("type", "A")
 .eq("severity", "High"))

A pre-built Query may be nested directly instead:

Query("OR").eq("status", "Open").and_(Query().eq("type", "A").eq("severity", "High"))
Parameters:

query – Optional pre-built Query to nest as an AND group. If None, creates a fresh AND context for inline building.

Returns:

If query is provided, returns self (for chaining). Otherwise, returns a fresh AND-context proxy that appends to this Query when a terminal method (to_body/model) is called.

sort(field: str, direction: str = 'DESC') → Query[source]¶

Add a sort clause. Call multiple times for multi-field sort.

Parameters:

  • field – The field path to sort on (e.g. "createDate", "name").

  • direction – "DESC" (newest/highest first, default) or "ASC".

Query().sort("createDate", "DESC").sort("name", "ASC")
select(*fields: str) → Query[source]¶

Allowlist the fields returned per record (__selectFields).

Reduces payload size when you only need a few fields. Mutually exclusive with ignore():

Query().select("uuid", "name", "severity", "status")
ignore(*fields: str) → Query[source]¶

Denylist fields stripped from each record (__ignoreFields).

Useful when the module has large text fields you don’t need. Mutually exclusive with select():

Query().ignore("description", "sourcedata")
limit(n: int) → Query[source]¶

Set the page size (number of records per page).

The default server page size is 30. Use with page() for manual pagination, or leave it to iterate() which handles paging automatically:

Query().limit(100)          # 100 per page
page(n: int) → Query[source]¶

Set the 1-based page number for manual pagination.

Usually not needed — prefer iterate() which walks pages automatically:

Query().limit(50).page(3)   # third page of 50
search(term: str) → Query[source]¶

Add a free-text search term alongside structured filters.

The server applies full-text search across indexed fields in addition to any leaf filters already added:

Query().eq("status.itemValue", "Open").search("ransomware")
model() → pyfsr.query_models.QueryBody[source]¶

Return the validated, typed QueryBody for this query.

pyfsr.appliance¶

Ergonomic Python facade for the pyfsr appliance operations.

The appliance verbs (DB queries, service control, queue management, log tails, Elasticsearch/HA health, …) are implemented as plain functions under pyfsr.cli.appliance, grouped per module — fine for the CLI, awkward to call by hand.

Appliance wraps a single connection and exposes those verbs as grouped methods (appliance.db.query(...), appliance.service.status(), appliance.mq.queues()) so the Python API mirrors the CLI:

>>> from pyfsr import Appliance
>>> box = Appliance(host="10.0.0.1", user="csadmin", key_path="~/.ssh/id_rsa")
>>> box.info()
{'target': 'csadmin@10.0.0.1', 'fsr_version': '7.6.0', 'content_db': 'venom',
 'device_uuid': '0123…cdef', 'db_user': 'cyberpgsql'}
>>> dbname, headers, rows = box.db.query("SELECT count(*) FROM alerts")
>>> (dbname, headers, rows)
('venom', ['count'], [['42']])

Or reuse a client’s host (SSH still needs its own credentials):

>>> from pyfsr import FortiSOAR
>>> client = FortiSOAR("https://10.0.0.1", token="...")
>>> box = client.appliance(key_path="~/.ssh/id_rsa")

The same --yes / --write gating as the CLI applies: mutating methods take yes=True and SQL writes go through box.db.execute(..., yes=True). For any command not surfaced as a named method, Appliance.run() is the escape hatch.

class pyfsr.appliance.Appliance(host: str | None = None, *, user: str = 'csadmin', password: str | None = None, port: int = 22, key_path: str | None = None, sudo_password: str | None = None, insecure_skip_host_key_check: bool = False, instance: str | None = None, _facts: pyfsr.cli.appliance.facts.Facts | None = None)[source]¶

A connection to a FortiSOAR appliance, exposing the pyfsr appliance verbs.

Construct it with SSH connection details (or run it on-box with no host to use a local transport), then reach the grouped verbs:

  • db — Postgres queries, table cleanup

  • service — start/stop/restart, liveness

  • mq — RabbitMQ queues, purges

  • host — memory/load/disk/RSS

  • license — device UUID, drift

  • logs — tail, scan, bundle

  • es — Elasticsearch health/shards

  • ha — cluster nodes/health/replication

  • certs — TLS cert regeneration

plus info(), diagnose(), and run() (the escape hatch for arbitrary shell commands).

Connection args fall back to PYFSR_APPLIANCE_HOST / _USER / _PASSWORD when omitted. Pass instance="<alias>" to resolve a named SSH profile from ~/.pyfsr/instances.toml (the same file InstanceRegistry uses for the REST client) — this takes precedence over the explicit host/user/password kwargs and is the SDK counterpart of pyfsr appliance --instance <alias>.

Note

Each grouped verb returns a typed result defined in the corresponding pyfsr.cli.appliance submodule rather than here, so those dataclasses are documented on their own pages, not under this module’s contents:

  • pyfsr.cli.appliance.db — DatabaseInfo, DataClassSize, OrphanTable

  • pyfsr.cli.appliance.service — ServiceState, Listener, ProbeResult, ServiceActionResult

  • pyfsr.cli.appliance.mq — QueueInfo, Consumer, Permission, PurgeResult, WorkflowPurgeReport

  • pyfsr.cli.appliance.host — MemInfo, LoadAvg, ProcRss, DiskUsage, HostSnapshot

  • pyfsr.cli.appliance.license — LicenseDetails, DriftReport

  • pyfsr.cli.appliance.es — ESHealth

  • pyfsr.cli.appliance.ha — HaNode, HaHealth

run(argv: list[str], *, input_text: str | None = None, env: dict[str, str] | None = None, timeout: float = 60.0, sudo: bool = False) → str[source]¶

Run an arbitrary command on the appliance; returns stdout.

Raises RuntimeError on non-zero exit. Use this as the escape hatch for commands not covered by a namespace method.

>>> box = Appliance(host="10.0.0.1", key_path="~/.ssh/id_rsa")
>>> box.run(["uname", "-r"])
'5.15.0-101-generic\n'

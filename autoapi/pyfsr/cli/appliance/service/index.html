pyfsr.cli.appliance.service¶

pyfsr appliance service — systemd / cyops service verbs.

Wraps csadm services plus a liveness probe that catches the failure systemctl status misses: a service that is active but wedged (accepting connections but never responding). The probe curls canonical endpoints on-box with a hard timeout; HTTP 000 / a hang ⇒ “active but wedged → restart”.

Free-form, ANSI-coloured text — for a typed result use services().

NB: csadm services --status has no --name flag; passing one is silently ignored and the full list comes back (live-verified). So name filters the lines client-side here rather than being handed to csadm.

pyfsr.cli.appliance.service.restart(transport: pyfsr.cli.appliance.transport.Transport, name: str, *, yes: bool = False) → ServiceActionResult[source]¶

Restart a single cyops service via csadm services --restart-service. Gated by yes.

Use this — NOT restart_all() — for one service. The bare --restart flag (restart_all) bounces the WHOLE stack in order; --restart-service touches just name (stop → sleep 5 → start, per csadm).

csadm exits 0 even when it refuses an unknown/unmodifiable name (it prints an ERROR: ... can not be modified hint and no-ops), so the returned ok folds that rejection text in — a typo’d name yields ok=False, not a false success. (live-verified, see _CSADM_REJECT.)

pyfsr.cli.appliance.service.stop(transport: pyfsr.cli.appliance.transport.Transport, name: str, *, yes: bool = False) → ServiceActionResult[source]¶

Stop a single cyops service via csadm services --stop-service. Gated by yes.

Used to quiesce a worker (e.g. celeryd) so a queue can be purged without it re-draining mid-operation — see pyfsr.cli.appliance.mq.purge_workflows(). Like restart(), a rejected name surfaces as ok=False despite csadm’s exit 0.

pyfsr.cli.appliance.service.restart_all(transport: pyfsr.cli.appliance.transport.Transport, *, yes: bool = False) → ServiceActionResult[source]¶

Restart the WHOLE service stack in order via csadm services --restart. Gated by yes.

This is the full-stack bounce (stop-all → start-all), distinct from restart() which restarts a single named service. Disruptive — every service drops — so it is gated behind yes.

pyfsr.cli.appliance.service.systemctl(transport: pyfsr.cli.appliance.transport.Transport, action: str, unit: str, *, signal: str | None = None, yes: bool = False) → ServiceActionResult[source]¶

Drive systemd directly: systemctl <action> <unit> (sudo).

This is the forceful path that bypasses csadm orchestration — use it when a unit is wedged and the graceful stop()/restart() won’t take. kill sends SIGTERM by default; pass signal (e.g. "SIGKILL", "9") to escalate. Mutating actions are gated by yes; read-only ones (status, is-active, show) run unconditionally.

Returns a ServiceActionResult; for read-only actions output carries the queried value (e.g. "active" from is-active). Note unit is the systemd unit name (e.g. celeryd.service), not always the csadm label.
